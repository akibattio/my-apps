# CLAUDE.md — ソフコミ 広告運用 全社共通ルールブック

> このファイルは「会社としての判断作法」をシステム化したもの。
> Claude Code / AI は**毎回このルールに従って**分析・提案を行う。担当者が代わっても判断がブレないための最上位ドキュメント。
> 個社固有の値（目標CPA・接続情報など）は `clients/<社名>.md` に置く。**個社MDの記載が本ファイルより優先**される（下記「2階建て運用」参照）。
>
> 最終更新：2026-07-02 ／ 管理：ソフトコミュニケーションズ

---

## 0. 絶対原則（これだけは絶対に守る）

1. **書き込みは人間の承認後のみ。** auto-approval は使わない。AIは「提案の下書き」までしか作らない。
2. **新規エンティティは必ず PAUSED で下書き生成。** 承認者が明示的に有効化する。
3. **トークン・シークレットの実値は絶対にコード/MD/チャットに出さない。** 参照は環境変数名のみ。
4. **不確かなら止めて確認する。** 数値の根拠が薄い・データ欠損・前提が崩れている場合は、提案せずエスカレーション。
5. **媒体の広告ガイドライン（特に医療・美容）を最優先。** 効果保証・誇大表現は提案段階で排除する。

---

## 1. 2階建て運用（このルールブックと個社MDの関係）

- **全社共通（本ファイル）**：判断作法・禁止事項・レポート様式・承認フロー・エスカレーション。
- **個社（`clients/<社名>.md`）**：接続情報（環境変数名のみ）・目標CPA/ROAS・悪化しきい値・除外方針・トーン&マナー・広告主固有の禁止事項。

**優先順位**：個社MDに具体値がある場合はそれを使う。無い場合のみ本ファイルの既定値を使う。個社MDと本ファイルが矛盾する場合は、**より厳しい方**を採用し、矛盾自体をレポートに明記する。

---

## 2. 判断基準（しきい値）

> ⚠️ 以下は**全社の既定値（プレースホルダ）**。実運用の数値は個社MDで必ず上書きすること。
> 数値の根拠が個社MDに無い場合、AIは提案に「（全社既定値を使用・要確認）」と付記する。

### 2.1 パフォーマンス判定

| 指標 | 既定のしきい値 | アクション |
|---|---|---|
| CPA が目標CPAを **+20%** 超過（7日移動平均） | 悪化 | 予算減 or 入札調整を提案 |
| CPA が目標CPAを **+50%** 超過 | 重度悪化 | 停止候補として提案＋アラート |
| ROAS が目標を **-20%** 下回る | 悪化 | 配信面/クリエイティブ見直しを提案 |
| CV=0 かつ 消化額が「日予算×3」超 | 無駄消化 | 停止候補として提案＋アラート |
| CTR が同キャンペーン平均の **50%未満** | クリエイティブ疲弊の疑い | 差し替え提案 |
| フリークエンシー **3.5 超**（Meta） | 疲弊/枯渇 | オーディエンス拡大 or クリエイティブ更新を提案 |

- 評価期間の既定：**直近7日**（曜日変動をならすため移動平均を使う）。判断が割れる時は 7日/14日/28日を併記。
- **学習期間中（配信開始から一定日数・CV数未達）のエンティティは判定対象外**とし、その旨を明記（早すぎる判断を避ける）。

### 2.2 予算変更のガード（額しきい値）

| 変更幅（日予算比） | 承認レベル |
|---|---|
| ±20% 以内 | 担当者承認 |
| ±20% 超 | 上長承認 |
| 大型クライアント（月額 100万円〜） | 二段階承認（担当＋レビュア）を常時必須 |

- 1回の提案での増額は原則 **+30%/回** を上限（急激な増額は学習リセット・浪費リスク）。
- 減額・停止方向は上記より緩めて良いが、**全停止**は必ず上長承認。

## 3. 禁止事項・コンプライアンス

### 3.1 広告表現（全業種共通）

- 効果・効能の**保証／断定**（「必ず」「100%」「絶対」「No.1」根拠なし）を提案文・クリエイティブ案に入れない。
- 誇大表現・比較優良誤認・最上級表現（根拠のない「最安」「業界唯一」）を避ける。
- ビフォーアフター/体験談は媒体規定に従い、根拠・出典・打消し表示の要否を確認。

### 3.2 医療・美容（ハルナ美容外科等で特に重要）

- **医療広告ガイドライン（厚労省）**および各媒体（Meta/Google）の医療・美容カテゴリ規定に準拠。
- 禁止・要注意：効果の保証、術前術後写真の不適切な使用、体験談による効果訴求、費用の誤認を招く表示、比較優良・最上級表現。
- 限定解除要件（自由診療の場合の治療内容・費用・リスク・副作用の明示）を満たさない訴求は提案しない。
- **判断に迷う表現は必ず人間（薬機・医療広告の確認者）にエスカレーション**。AIが「大丈夫」と断定しない。

### 3.3 データ・運用

- 個人アカウント依存の接続を作らない（組織所有のSystemUser/サービスアカウントに寄せる）。
- クライアント間でトークン・アカウントを混在させない。操作ログは名義で追える状態を保つ。

## 4. 承認フロー（承認ゲート）

```
毎朝 Routines 実行
  → データ取得（読み取りのみ）
  → 本ルール＋個社MDで分析
  → 提案を下書き生成（新規は PAUSED）
  → 承認キューに積む（管理画面 / 通知）
  → 人間が承認/却下
  → 承認分のみ適用（書き込み）
  → 適用結果をログ＋レポートに記録
```

- **却下も理由を記録**（次回以降の提案精度・ルール改善に使う）。
- 承認待ちが一定時間（既定：営業日で24h）滞留したらリマインド。
- 書き込みは「誰が・いつ・何を・変更前後の値」を必ずログ化。

## 5. エスカレーション手順

| 状況 | 対応 |
|---|---|
| 医療・薬機表現の可否判断 | 医療広告確認者へ即エスカレーション（AI判断で確定しない） |
| 日予算±20%超 / 全停止 | 上長承認 |
| 重度悪化アラート（CPA+50%・無駄消化） | 担当へ即通知、当日中に対応方針を決める |
| データ取得失敗・数値欠損・API異常 | 提案を止め、接続担当へ連絡（推測で提案しない） |
| 大型クライアントの重要変更 | 二段階承認（担当＋レビュア） |

## 6. レポート様式（標準フォーマット）

すべての分析アウトプット（日次/週次/月次）は以下の順で構成する：

1. **サマリー**（3行以内）：全体の良し悪し・要対応件数・特記事項。
2. **KPIハイライト**：期間の 費用 / CV / CPA / ROAS を、目標・前期間と対比（増減率付き）。
3. **要対応**：しきい値に抵触した項目を重要度順。各項目に「事実 → 推定原因 → 提案アクション → 承認レベル」。
4. **提案（下書き）一覧**：適用すると何がどう変わるか（変更前後の値）。すべて未適用であることを明記。
5. **注記**：使用データ期間・学習中で除外した項目・全社既定値を使った箇所・要確認事項。

- 数値は必ず**期間・母数**を添える（「CPA悪化」ではなく「直近7日CPA ¥12,000（目標¥8,000、+50%）」）。
- 断定を避け、根拠と信頼度を示す。推測には「推定」と明記。

---

## 7. トーン&マナー（提案文・レポート文）

- 事実ベース・簡潔・専門用語は必要最小限（引き継ぎやすさ最優先）。
- クライアント提出物は敬体、社内メモは常体で可。個社MDに指定があればそれに従う。
- 断定・煽り表現を避け、「〜と考えられます／提案します」の形。

---

## 8. セキュリティ（詳細は各MDと `.env.example`）

- トークン/シークレットの実値はリポジトリ・MD・チャットに出さない（**環境変数名のみ**記載）。
- Meta の一時トークンは60日失効 → 無人運用は組織所有の System User Token へ。
- クライアントごとにトークン分離。Git に `.env`・secrets を絶対コミットしない（`.gitignore` で除外）。

---

## 9. AIへの運用指示（このリポジトリでの振る舞い）

- 提案生成時は必ず **本ファイル → 該当 `clients/<社名>.md` の順で読み込み**、個社値を優先する。
- **書き込み系API（予算変更・入札変更・ON/OFF・作成）を自発的に実行しない。** 生成物は常に「下書き」。
- 数値を扱うときは期間・母数・信頼度を明示。データが無い項目は「データなし」と書き、埋めない。
- しきい値抵触は必ずレポートの「要対応」に、承認レベルとともに載せる。
- 迷ったら §5 に従いエスカレーション。**沈黙して自己判断で進めない。**
