## 1. プロジェクトのゴール

Google 広告と Meta（FB/Instagram）広告の運用を **半自動化** し、クライアントワークで使う。

- **自動化する範囲**：データ取得 → 分析 → 施策の「提案・下書き」生成 → レポート
- **人が担う範囲**：提案の承認/却下（＝書き込みは承認後にのみ実行。無人化はしない）
- **最重要方針：属人化しない**。担当が代わっても・経験が浅くても、トラブルなく引き継げる運用基盤にする。
- 将来は月額100〜300万円規模の大型クライアントにも対応（二段階承認・変更額しきい値などのガードを追加）。

---

## 2. 全体設計

### 非属人化の3資産（人の頭の中をシステムへ移す）
1. **判断基準 → CLAUDE.md ルールブック**：CPA基準・悪化しきい値・除外方針・トーン&マナー。AIが毎回同じ基準で提案する。
2. **接続情報 → 接続台帳MD**：どのアカウントに・どの権限で・どのトークンで繋いでいるか。個人アカウント依存を排除。
3. **手順 → SOP/プレイブック**：新規立ち上げ・月次レポート・トラブル対応をスキル化。

### 運用ループ（毎朝 Routines で自動実行）
データ取得 → ルールブックで分析 → 提案を下書き生成（新規はPAUSED） → 人が承認 → 承認分のみ適用

### 承認ゲート（必須）
- すべての書き込みは人間の承認後にのみ実行。**auto-approval は使わない**。
- 大型クライアントは二段階承認（担当＋レビュア）、日予算 ±20% 超は上長承認。

### 通知（オプション）
- Slack 連携は任意。要対応アラートと承認待ちを指定チャンネルに投稿。不要なら管理画面のみで運用。

## 3. リポジトリ構成（推奨）

```
sofcom-ads/
├── HANDOFF_広告運用自動化.md    ← このファイル
├── CLAUDE.md                     ← 全社共通ルール（未作成・最優先で作る）
├── clients/
│   └── ハルナ美容外科.md         ← 個社の接続台帳＋ルール（雛形あり）
├── skills/                       ← 監査・レポート等のスキル（Claude Ads を土台に）
├── routines/                     ← 定期実行ジョブの定義
├── console/                      ← 管理画面（プロトタイプ ad_ops_console.jsx が土台）
└── .env / secrets（Gitには入れない）
```

- 御社の社内展開構想 `sofcom-claude/` と命名を揃える。
- **2階建て運用**：全社共通 `CLAUDE.md` ＋ `clients/<社名>.md`。

## 6. 媒体別セットアップ要件

### Meta（先に着手・当日で動く）
1. Meta Business Manager で対象クライアントの広告アカウント（`act_xxxx`）にアクセスできる状態か確認。
   - ※ 既に運用中なのでアクセス済みの可能性が高い。まずここを確認。
   - 無ければクライアントに「パートナー追加／広告管理者権限付与」を依頼。
2. Meta 公式 Ads コネクタ（MCP/CLI）を接続。
   - Claude Code：`pip install meta-ads`（Python 3.12+）
   - 認証は Meta Business OAuth のクリックのみ。**Developer App 審査は不要**。
3. まず読み取りで疎通確認（例：「先週のROAS」「CPAが高い広告セット」）。
4. 無人運用（Routines）まで行くなら **無期限の System User Token**（Business Manager で発行、60日失効の罠を回避）。テスト段階は手動OAuthで可。

**必要情報**：Business Manager アクセス／クライアントの広告アカウントID（`act_xxxx`）／（無人化するなら）System User Token。

### Google（申請だけ先行・審査数日〜2週間）
1. **APIセンターで開発者トークンを申請**：https://ads.google.com/aw/apicenter
   - MCC（クライアントセンター）アカウントの管理者でログイン。個人管理アカウントでは発行不可。
   - API Access フォーム記入＋規約同意。会社サイトURLが有効であること、連絡先メールが監視されていること。
   - 用途欄：「自動レポート・分析・運用最適化のためのMCP/API連携」。
   - 発行直後は「テストアカウント」レベル → **ベーシックアクセスを申請**して本番実アカウントを解禁。
2. Google Cloud プロジェクト作成 → 課金有効化 → Google Ads API 有効化。
3. OAuth 2.0 クライアントID（デスクトップアプリ型）作成 → Client ID / Secret 取得 → Refresh Token 生成。
4. 読み取りは Google 公式 MCP（`pipx` で導入、Python 3.12+）。**書き込みは公式MCPに無い** → 自前 Google Ads API か Adspirer 等のホスト型。

**必要情報**：MCCのカスタマーID（`LOGIN_CUSTOMER_ID`）／クライアントのカスタマーID（10桁・ハイフン無し）／開発者トークン／Client ID／Client Secret／Refresh Token。

## 8. 最初のタスク（Claude Code はこの順で）

1. リポジトリ `sofcom-ads/` を初期化し、このファイルと `clients/ハルナ美容外科.md`（雛形）を配置。
2. **全社共通 `CLAUDE.md`（ルールブック）を作成**：会社としての判断作法・禁止事項（誇大表現・医療広告ガイドライン等）・レポート様式・承認ルール・エスカレーション手順。※このプロジェクト最優先の成果物。
3. Meta CLI/MCP を接続し、テスト1社で読み取り疎通確認。
4. しきい値照合 → 提案生成のプロンプト/スクリプトを作成（提案は下書きのみ、書き込みしない）。
5. Slack通知（任意）を組むか判断。
6. Google 承認が下りたら公式MCPで接続し、読み取りから追加。
7. 管理画面（`ad_ops_console.jsx`）を実データに接続する設計に着手（バックエンド＋定期取得）。

## 10. セキュリティ・注意事項

- トークン/パスワードの実値はリポジトリ・MD に置かない（環境変数名のみ）。
- 書き込みは必ず人間承認。新規エンティティは PAUSED 生成。
- クライアントごとにトークン分離・操作ログの名義管理。
- Meta の一時トークンは60日で失効 → System User Token（無期限・組織所有）へ。
- 医療・美容などは各媒体の広告ガイドライン（表現制限）に準拠。
